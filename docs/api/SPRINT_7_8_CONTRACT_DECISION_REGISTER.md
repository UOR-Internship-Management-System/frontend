# Sprint 7-8 OpenAPI Contract Decision Register

Contract: `CV_Management_API_OpenAPI_v1.6.0.yaml`
Status: Final contract-freeze decisions for frontend/backend implementation planning

## Authority applied

1. Final Reduced Scope Baseline v1.1 and Scope Reductions
2. SRS v3.0.1
3. Backend Module Documentation and Database Design Document
4. Production-Ready Use Cases and Student/Admin Workflows
5. UI Frontend Specification
6. API specification and existing OpenAPI conventions
7. Repository plans, Admin source description, HTML wireframes, and DESIGN.md

Wireframes were treated as layout/interaction evidence, not as permission to restore removed behavior.

## Decisions

| ID   | Area                         | Final decision                                                                                                                                                                                 | Reason and scope effect                                                                                                                                                                              |
| ---- | ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| D-01 | Version                      | Freeze as OpenAPI `1.6.0`.                                                                                                                                                                     | Sprint 7-8 adds and hardens public DTOs, lifecycle enums, constrained queries, file formats, and concurrency behavior while preserving prior paths and operationIds.                                 |
| D-02 | Student deep-dive shape      | Keep declared skills, projects, and academic records as separate paged resources; return identity/profile/CV-supporting summary and latest-CV availability from the top-level detail resource. | Supports partial loading and avoids one large unbounded response while preserving existing paths.                                                                                                    |
| D-03 | Latest saved CV absent       | Metadata endpoint returns `200` with `availability: NOT_SAVED`; download returns the existing no-saved-CV Problem Details response.                                                            | Gives the UI a stable neutral state without inventing submission/review status.                                                                                                                      |
| D-04 | Company lifecycle            | `DELETE /admin/companies/{companyId}` performs idempotent soft deactivation. Existing references remain; active requests can cause `409`.                                                      | Aligns with `is_active` in the database design and avoids destructive loss of request history.                                                                                                       |
| D-05 | Company fields               | Freeze name, website URL, contact person/email/phone, notes, active state, timestamps, and version. Do not include industry or account/login fields.                                           | Website appears in the approved UI evidence and frontend URL-validation guidance; industry lacks sufficient authority. Backend persistence must add a nullable website column or equivalent mapping. |
| D-06 | Internship request lifecycle | Use `DRAFT`, `ACTIVE`, `CLOSED`, `CANCELLED`. `DELETE` idempotently moves DRAFT/ACTIVE to CANCELLED.                                                                                           | Matches the Database Design Document; avoids inventing an ARCHIVED state.                                                                                                                            |
| D-07 | Internship request fields    | Freeze company, role title, optional description/location/work mode, status, optional advisory shortlist guidance, notes, required skills, timestamps, and version. Omit deadline.             | Deadline is not present in the controlling database/workflow requirements. GPA is explicitly prohibited.                                                                                             |
| D-08 | Request skill mutation       | Create/update may atomically set or replace the full required-skill set. Nested endpoints remain for incremental add/remove edits.                                                             | Removes ambiguity: omitted skills mean unchanged on PATCH; present skills mean complete replacement. Nested mutations are individually concurrency-protected.                                        |
| D-09 | Competency requirement       | A request skill may carry a nullable required competency level.                                                                                                                                | Explicitly supported by the physical database design; no runtime competency-level array is added to filtering criteria.                                                                              |
| D-10 | Filtering runs               | Persist sanitized runtime criteria and a count, but recompute candidate rows from latest committed official GPA and current declared skills when results are read.                             | Existing GET-by-run path implies persistence; recomputation preserves the “latest committed data” rule.                                                                                              |
| D-11 | Runtime filtering criteria   | Use nullable min/max GPA, selected request-skill IDs, additional taxonomy skill IDs, and explicit AND/OR mode.                                                                                 | GPA remains runtime-only and does not mutate the internship request. No score/rank fields are introduced.                                                                                            |
| D-12 | Students without GPA         | Exclude when any runtime GPA bound is active; otherwise they may appear with `officialGpa: null` and explicit availability status.                                                             | Provides deterministic behavior and a representable neutral state.                                                                                                                                   |
| D-13 | Candidate warning            | Return factual `hasExistingActiveShortlist` and count metadata.                                                                                                                                | Cross-shortlist history is non-blocking decision support, not an eligibility conflict.                                                                                                               |
| D-14 | Shortlist cardinality        | One shortlist resource per internship request in Version 1. Duplicate creation returns `409`.                                                                                                  | Matches the simplified locked scope and active-request workflow; no historical re-shortlisting workflow is authorized.                                                                               |
| D-15 | Candidate selection          | Candidate selection is explicit manual mutation. Batch-add is supported for the filtering UI; duplicates are reported rather than duplicated.                                                  | Supports multi-row manual selection without automated selection.                                                                                                                                     |
| D-16 | Guidance behavior            | Guidance is advisory. If exceeded, finalization requires explicit acknowledgement but remains permitted.                                                                                       | Enforces the non-blocking requirement and avoids hard capacity validation.                                                                                                                           |
| D-17 | Concurrency                  | Use quoted integer `If-Match` values and ETag responses for company, request, draft shortlist mutation, and finalization. Missing precondition returns `428`; stale state returns `412`.       | Consistent with existing repository conventions and optimistic-lock guidance.                                                                                                                        |
| D-18 | Finalization                 | Only DRAFT can atomically become FINALIZED; finalized candidates are immutable; repeat/concurrent operations return state or version conflict.                                                 | Protects multi-Admin finalization and audit integrity.                                                                                                                                               |
| D-19 | Summary export               | Asynchronous CSV only.                                                                                                                                                                         | Separates one approved export purpose and freezes exact media type/filename behavior.                                                                                                                |
| D-20 | Bulk CV export               | Asynchronous ZIP containing available latest-saved ATS-compliant PDF CVs.                                                                                                                      | Exact format matches bulk-file delivery without exposing LaTeX source or CV history.                                                                                                                 |
| D-21 | Missing CVs                  | Partial bulk export may complete with explicit missing counts/list and warnings; zero available CVs fails safely.                                                                              | Missing files are never silently omitted. Shortlist state is unchanged.                                                                                                                              |
| D-22 | Export status                | Use `QUEUED`, `PROCESSING`, `COMPLETED`, `FAILED`, `CANCELLED`.                                                                                                                                | Matches the Database Design Document. URLs stay null until ready.                                                                                                                                    |
| D-23 | File security                | All downloads are Admin-only, audited, `no-store`, attachment-only, and expose no storage path.                                                                                                | Preserves Student privacy and file authorization requirements.                                                                                                                                       |

## Non-blocking implementation alignments

- Add nullable company website persistence because the UI contract retains `websiteUrl` while the current physical column catalogue does not list it.
- Final export retention duration remains deployment/configuration policy. The contract therefore keeps `expiresAt` nullable rather than inventing a duration.

Neither item changes Version 1 functional scope.
